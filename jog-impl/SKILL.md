# jog-impl: 実装〜レビューループスキル

Ready 状態の Issue を受け取り、実装計画の策定・コーディング・agent レビュー・
フィードバック対応を繰り返し、完了をユーザーに報告するまでを担う。

---

## ステップ 0: jog ツールのロード

```
ToolSearch: "jog craft"
```

---

## Phase 1: 前提確認

### 入力パターン

**Craft ID が指定されている場合**
`craft_get(id=N)` で Craft と関連 Issue を確認する。
既に着手中の Craft であれば、まず `actionable_feedback(craft_id=N)` で未対応フィードバックを確認してから作業を始める。

**Issue ID が指定されている場合**
1. `issue_get(issue_id=N)` で Issue を確認する
2. status が `ready` でなければユーザーに確認する（Ready 化は `jog-start` スキルの役割）
3. Issue に紐づく Craft がなければ Phase 2 の最初で作成する

**何も指定されていない場合**
現在のブランチから推測するか、ユーザーに「Craft または Issue の番号を教えてください」と尋ねる。

---

## Phase 2: Craft の準備

### Craft がまだない場合

```bash
git branch --show-current  # ベースブランチを確認
```

```
craft_create_from_issue(
  issue_id=N,
  title="<issue title と同じで OK>",
  body="",
  base="<現在のブランチ名>",
  create_worktree=false
)
```

worktree が必要かはユーザーに確認する。

### 実装計画を Craft body に記載する

Issue の内容・コメントを読んで実装計画を立てる。
`craft_get` で現在の body を取得し、以下の形式で更新する：

**既存の body がある場合:**
```
## 実装計画

<今回の計画をここに書く>

---

> 以下は計画策定前の記述（参考）

<既存の内容をそのまま残す>
```

**既存の body が空の場合:**
```
## 実装計画

<計画をここに書く>
```

計画には以下を含める：
- 実装するファイル・クラス・メソッドの概要
- 影響範囲と注意点
- テスト方針

---

## Phase 3: 実装

計画に沿って実装を進める。

- 不明点はユーザーに確認して構わない
- 実装中に計画を修正した場合は Craft body も更新する
- コミットのタイミングはユーザーの指示に従う
    - コミットは、指示がなければ論理的な単位で行う

---

## Phase 4: レビューループ

実装が終わったら、以下のループを繰り返す。

### 4-1: レビュー agent に依頼

Agent ツールで以下のプロンプトを使ってレビューを依頼する：

```
あなたはコードレビュアーです。Craft #<N> の実装をレビューしてください。

## 制約（必ず守ること）
- Jog MCP を使用すること（ToolSearch で "jog" を検索してロードする）
- フィードバックは必ず Craft #<N> に `comment_add` でコメントとして残すこと
- コードに対するコメントでは、code_refを指定すること
- 関連コメントがある場合は、reply_toを指定すること
- 問題がない場合も「レビュー完了。問題なし。」とコメントを残すこと

## レビュー手順
1. `craft_get(id=<N>)` で Craft の概要と実装計画を確認する
2. `craft_diff(id=<N>, scope="full")` で変更差分を確認する
3. 関連 Issue のコメントも読んで、要件を満たしているか確認する
4. 以下の観点でレビューする：
   - 要件・仕様を満たしているか
   - バグや論理的な誤りがないか
   - コード品質・設計上の懸念点
5. フィードバックを `comment_add` でコメントに残す（1件ずつ、具体的に）
6. 問題がなければ「レビュー完了。問題なし。」とコメントを残す
```

### 4-2: フィードバックの確認

agent 完了後、`actionable_feedback(craft_id=<N>)` を呼び出す。

- **結果が空（またはアクションすべき項目なし）** → Phase 5 へ
- **アクションすべき項目あり** → 4-3 へ

### 4-3: フィードバックへの対応

`actionable_feedback` が返したフィードバックを順に対応する。

- 基本的に自律的に対応する
- 判断が難しいもの（仕様変更を伴う、影響範囲が大きい）はユーザーに確認する
    - 同時に、comment_addでreply_to付きのコメントを残す
- 1つのフィードバックへの対応が終わったら**即座に** `comment_resolve` で resolved にする
    - 後でまとめて resolve しようとすると積み残しになりやすいため

### 4-4: 対応内容のサマライズとコメント

全フィードバックへの対応が終わったら：

1. 対応した内容を要約して Craft にコメントを追加する

```
## フィードバック対応サマリー（<日付>）

対応したフィードバック：
- [コメントID or 概要] → 対応内容
- [コメントID or 概要] → 対応内容
...

見送ったもの（理由）：
- なし / [概要] → 理由
```

### 4-5: ループ判定

4-1 に戻り、再度 agent レビューを依頼する。
`actionable_feedback` が空になるまで繰り返す。

---

## Phase 5: 完了報告

ユーザーに以下を報告する：

```
## 実装完了

| 項目 | 内容 |
|------|------|
| Issue | #N「〜」 |
| Craft | #M (feat/00N-...) |
| レビュー回数 | X 回 |

### 実装内容の概要
<何を実装したか 3〜5行>

### レビューで対応したこと
<対応内容の要点>
```

**Craft の merge はユーザーから明示的に指示されない限り行わない。**

---

## やってはいけないこと

- Issue が Ready でないのに実装を始める
- `craft_merge` をユーザーの指示なしに実行する
- フィードバック対応のサマリーコメントを残さずに resolved にする
- `actionable_feedback` を確認せずにループを終了する
